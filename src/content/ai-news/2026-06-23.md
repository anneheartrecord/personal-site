## 今日判断

我今天看下来，真正有用的信息几乎都在两条线上：一条是模型开始吞掉一部分原本写在 harness 里的调度逻辑，另一条是产品团队越来越老实地回到上下文、权限、审计这些脏活累活。前者的代表是多模型路由和 Logan 那句 model eats the harness，后者的代表是 Aaron 讲的 headless software。我的判断是，2026 年做 Agent 已经不是拼谁先把工具调用串起来，而是拼谁能把路由、验证、上下文和权限这四件事做成默认能力，最好还别把复杂度暴露给开发者。

另外我更在意那些带具体机制的工作流判断，而不是泛泛而谈。比如做视频 agent 的团队最后退回 HTML/CSS/JS，不是因为浏览器老派，而是因为 LLM 对代码结构的可控性远高于对视觉细节的直接理解；再比如 Zara 那条关于 context 长度的经验，我觉得很像这轮产品落地的真实分水岭：输出质量差，很多时候不是模型不够强，而是输入没有把任务约束、材料和判断标准喂够。

如果把这些信息放在一起看，我的具体判断是：接下来一线 builder 最该投资的不是再套一层 Agent 概念，而是把 context engineering、模型路由、执行权限和人工接管点做细。谁能把这些机制藏到产品内部，让用户只感觉结果更稳、更快、更少返工，谁才更可能留下来。

### 1. Sakana 把多模型路由包装成单一 API

[查看原文](https://x.com/levie/status/2068917230570795178) · 来源：Aaron Levie (@levie)

Aaron 转述了 Sakana 新模型 Fugu 的思路：对开发者暴露一个 API，内部自动做模型选择、任务委派、结果验证和综合输出，必要时协调多个专家模型完成任务。它重要，不是因为多 Agent 这个词新，而是因为很多应用层团队本来就在私下这么搭 agent harness，现在开始被收敛成标准接口。我的判断是，多模型路由会继续从应用内部能力变成显式平台层，下一步竞争点会落在路由准确率、验证成本和对外隐藏复杂度的能力。

### 2. Aaron：Agent 会把企业软件推向 headless 使用

[查看原文](https://x.com/levie/status/2068851573175021864) · 来源：Aaron Levie (@levie)

Aaron 的判断很直接：当 Agent 使用软件的频次远超人类，软件价值会从界面交互转到 headless 执行能力，随之变重要的是 guardrails、审计日志、权威数据源和人机协作机制。他还点明一个常被低估的事实：一次 agentic 查询可能拉取的企业数据，比一个员工一个月接触的还多。为什么重要？因为这会改写企业软件的产品边界。我的判断是，下一阶段卖给企业的不是会聊天的 AI，而是能安全接管高频操作的软件底座。

### 3. Logan Kilpatrick 继续押注 model eats the harness

[查看原文](https://www.youtube.com/watch?v=cMAs8z2dehs) · 来源：Training Data

Training Data 这期和 Logan Kilpatrick 聊的核心命题是 model eats the harness。结合节目里关于原生多模态、实时编辑和 API builder 体验的讨论，Google 的意思已经很清楚：过去很多靠外部流程编排、模板和小技巧实现的能力，会逐步被更强的模型原生吸收。为什么重要？因为这影响所有做 Agent 和开发者工具的架构选择。我的判断是，外部 harness 不会消失，但会越来越像权限层、状态层和可靠性层，而不是智能本身。

### 4. 做视频 Agent 的团队发现 HTML 比纯视觉理解更可靠

[查看原文](https://x.com/petergyang/status/2068755908319236338) · 来源：Peter Yang (@petergyang)

Peter Yang 引用了一个做视频 agent 团队的复盘：他们原本想直接让 agent 理解视觉并生成视频，但踩坑后发现模型缺乏稳定的视觉智能，于是改用 HTML、CSS、JavaScript 作为中间表达，让 LLM 先生成结构和视觉规则，再把素材、图片、SVG 和镜头组合进去。为什么重要？这不是技巧，而是架构取舍。我的判断是，凡是要求高可控输出的多模态产品，都会优先寻找模型更擅长的中间语言，而不是硬赌端到端视觉生成。

### 5. Zara 给了一个避免 AI slop 的实用阈值

[查看原文](https://x.com/zarazhangrui/status/2068923768500793603) · 来源：Zara Zhang (@zarazhangrui)

Zara 提了一个很朴素但实用的经验法则：如果输入的 context 比输出短很多，结果大概率会变成 AI slop；她自己的经验是，高质量输出往往要求输入长度达到输出的 3 到 5 倍。为什么重要？因为很多团队把问题归咎于模型，实际上是上下文工程没做够。我的判断是，这条规则不该被当作 prompt 技巧，而该被写进产品设计里：好的 AI 功能必须主动帮用户补全背景、素材、限制条件和评价标准。
