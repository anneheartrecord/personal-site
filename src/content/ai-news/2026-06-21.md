## 今日判断

我今天更关注的不是谁又发了一个更强模型，而是 builder 开始把 agent 当成一个真正要安置在工作流里的执行者来设计。几个信号放在一起看很明显：Codex 这类工具被反复提到的，不再只是代码补全，而是浏览器、computer use、remote/local handoff 这种跨环境执行能力；另一边，大家讨论的也不是抽象智能，而是 agent 到底住在哪里、怎么和人交接上下文、用什么最小接口保持可监督。文件系统、Markdown、CLI 这些老东西重新变热，不是因为怀旧，而是因为它们足够稳定、可组合、对人和模型都可读。

我看下来，这波产品演进正在把很多团队从 API-first 拉到 workflow-first。以前大家先找 API 再拼自动化，现在已经有人直接让 agent 进浏览器、进电脑、进本地环境，把没有 API 的系统也纳入流程。这会让产品边界变宽，但也逼着开发者补齐另外几件事：更清晰的工作区、更好的文档结构、更可复现的 eval、更细的权限隔离。否则 agent 能做的越多，团队越难管。

我的判断是，接下来一线 builder 的分水岭不会先出现在模型参数，而会出现在谁更早把 agent 的工作介质定下来。谁能把 instructions、skills、logs、plans、artifacts 这些东西收进一个人和模型都读得懂的执行空间，谁就更容易做出真正可持续的 agent 产品。相反，只靠聊天框和花哨 demo 的，会越来越难跟真实工作流对接。

### 1. Codex 开始打通 remote 和 local handoff

[查看原文](https://x.com/thsottiaux/status/2068120572673077274) · 来源：Thibault Sottiaux (@thsottiaux)

Thibault 提到 Codex 已经支持 remote / local handoff，意思是 agent 不再被锁死在单一运行环境里，可以在远端和本地之间接力完成任务。这个变化重要，不是因为又多了一个功能点，而是因为它把 coding agent 从单机助手推向了真正的执行系统：代码、浏览器、shell、文件和权限边界开始被串起来。我今天的判断是，这类 handoff 能直接减少团队为了“让模型动起来”而额外搭的胶水层，但前提是产品把状态同步、权限控制和失败恢复做扎实，否则跨环境只是把复杂度藏起来。

### 2. 一线用户转向 Codex，原因不是聊天体验而是 browser 和 computer use

[查看原文](https://x.com/petergyang/status/2068175172960690266) · 来源：Peter Yang (@petergyang)

Peter Yang 说自己原本是重度 Claude Code 用户，但现在更常用 Codex，关键原因除了 GPT-5.5、速度和额度，更看重 browser 与 computer use。他给出的判断很实在：很多流程已经不用再到处找 API，而是直接依赖这两个能力完成。我觉得这条信息很有价值，因为它说明 builder 选工具的权重正在变化，聊天质量和前端品味仍重要，但能不能直接进入现有软件和网页完成任务，开始成为更硬的标准。我的判断是，未来 coding agent 的竞争会越来越像操作系统层竞争，不只是模型层竞争。

<a id="20260621-03"></a>

### 3. Aaron Levie 点明 agent 成败变量：共享工作区比更多提示词更重要

[查看原文](https://x.com/levie/status/2068068247413694532) · 来源：Aaron Levie (@levie)

Aaron Levie 这条我很认同。他说 agent 成功的首要变量，是能否拿到足够上下文；而上下文是否可用，很大程度取决于有没有一个人和 agent 都能理解的共享工作区。他进一步把这个工作区具象成文件系统里的 plans、notes、task lists、drafts、logs、decisions 这类材料。我认为这比很多“如何写 prompt”更有操作性，因为真实团队里最常见的问题不是模型不会答，而是任务状态没人和模型能共同维护。我的判断是，文件系统形态会成为 agent 产品的默认中间层，尤其适合需要监督、交接和回溯的任务。

### 4. Guillermo 把最小 agent 目录结构讲清楚：instructions.md 加 skills

[查看原文](https://x.com/rauchg/status/2068165988005380478) · 来源：Guillermo Rauch (@rauchg)

Guillermo 提出一个非常具体的 agent 打包方式：一个 `agent/` 目录，里面放 `instructions.md` 和 `skills/`，再用一条命令部署。他把 Markdown 说成下一代热门编程语言，听起来像玩笑，但背后其实是把 agent 定义压缩到最容易编辑、审查和版本管理的文本层。我看重的不是这个口号，而是这种结构让 agent 的行为定义更接近软件工件，而不是散落在聊天记录里。我的判断是，未来不少团队会把 agent 配置代码化，但未必先走 DSL 或复杂平台，反而会从 Markdown、目录约定和可部署模板开始。

<a id="20260621-05"></a>

### 5. Agent 正在倒逼软件重新尊重开放接口、文档和测试

[查看原文](https://x.com/rauchg/status/2067936390285807940) · 来源：Guillermo Rauch (@rauchg)

Guillermo 观察到，agent 正在推动一批原本就健康的软件习惯重新变成主流：开放 API、更好的文档、测试与 eval、CLI、支付协议，以及广泛接受 Markdown、JSON、HTML 这些格式。我觉得这不是泛趋势评论，而是对过去一年 builder 真实取舍的总结。只要你的系统要给 agent 用，那些不透明、只适合点点点的人类界面就会暴露问题。我的判断是，agent 时代不会先奖励最封闭的平台，而会先奖励那些接口清楚、文档完整、输出格式稳定的产品。

### 6. Anthropic 披露 Claude 在多产品里的 containment 方案

[查看原文](https://www.anthropic.com/engineering/how-we-contain-claude) · 来源：Anthropic Engineering

Anthropic Engineering 发布了关于如何在不同产品中对 Claude 做 containment 的工程文章。仅从标题也能看出重点不是模型能力，而是把模型放进真实产品后，如何约束它的权限、执行边界和风险面。这类内容对一线 builder 很重要，因为大家都在往 browser use、computer use、自动执行推进，能力一旦扩张，containment 就不是安全团队的附属话题，而是产品架构的一部分。我的判断是，接下来真正能规模化的 agent 产品，一定会把 containment 和 capability 一起设计，而不是等出事后补防护。
